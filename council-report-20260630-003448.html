Локальный пайплайн PNG → SVG: переработка концепта

5 советников · анонимное перекрёстное ревью · синтез председателя — 2026-06-30
Вопрос совету Переработать концепт «автоматического перевода PNG → SVG локальными инструментами» на MacBook Pro (заявлено: M5 Max, 128 ГБ RAM). Исходный план: LLM-оркестратор (Qwen Vision через Ollama) классифицирует картинку и выдаёт JSON-параметры → препроцессинг (OpenCV / ImageMagick / Real-ESRGAN) → трассировка VTracer → SVGO → валидация с авто-повтором. Нужен реалистичный и надёжный вариант.

Вердикт председателя

Где совет сходится

  • LLM-«мозг» не нужен. Все пятеро независимо: классификация «логотип / фото / эскиз» — это детерминированная эвристика на 5–6 признаках OpenCV (число уникальных цветов, доля краёв через Canny, энтропия гистограммы, доля прозрачности), а не Vision-модель. LLM вносит латентность, нагрев и галлюцинации туда, где нужна воспроизводимость.
  • Фото не трассируются. SVG — формат геометрии (контуры, Безье). Фотографию трассировка превращает в тысячи рваных полигонов — файл тяжелее исходного PNG. Честный ответ для фото: отказать или встроить растр через <image> base64.
  • Лишний overhead — выкинуть. Real-ESRGAN (плодит артефакты перед трассировкой), ComfyUI (это GUI для диффузии, не пайплайн-движок), LangGraph, «397B-модель». Ядро = VTracer + SVGO + bash/Python.
  • Ядро инструментов верное. VTracer и SVGO — правильный выбор; претензии только к обвязке.

Где совет расходится

  • Масштаб vs фундамент. Экспансионист хочет сразу батч-фабрику, MCP-сервер и само-тюнящийся пайплайн с обучением на SSIM. Остальные четверо: это автоматизация мусора поверх недоказанной базы. Разрешение: идея ценная, но строго после того, как ядро работает и появился честный триаж.
  • Роль LLM — ноль или почти ноль. Контрарианец/Первопринципник: убрать полностью. Исполнитель: допустимо позже как мелкий классификатор (llava 7b) для выбора пресета — но и тогда 3 руками подобранных пресета обычно бьют его.

Слепые пятна, пойманные на ревью

  • Нет критерия «хороший SVG». Без success-метрики (порог SSIM + бюджет path + размер) цикл авто-повтора не имеет условия остановки — у исходного концепта это потенциально бесконечный цикл.
  • Не задан вопрос «зачем вообще». Для логотипов и иконок исходник чаще уже существует в векторе (Figma / AI / PDF / шрифт). Реверс из растра решает проблему, которой во многих случаях быть не должно. Сначала спросить источник.
  • SSIM — ловушка как сигнал обучения. Он награждает буквальное копирование пикселей, то есть закрепляет фото-трассировку — ровно то, что нельзя делать.
  • Не учтены текст и градиенты как отдельные классы; не учтены объём/частота, приватность входа и человек-в-петле на масштабе.

Рекомендация

Соберите детерминированный конвейер из 6 этапов, без LLM в ядре. «Мозг» = метрики, а не нейросеть. LLM (мелкий vision-классификатор) — опциональный модуль третьей недели, и только если докажет, что бьёт пресеты на ваших данных.

1 · Триаж (метрики OpenCV) Считаем n_colors, edge-ratio (Canny), энтропию, долю alpha. Ветка vector-friendly / photo.
2 · Фото → честный fallback reject или обёртка <image> base64. Никакой трассировки и никакого Real-ESRGAN.
3 · Подготовка (только нужное) magick -fuzz 10% -trim +repage + квантование цветов по n_colors. Денойз/апскейл — точечно, не превентивно.
4 · Трассировка VTracer Пресет по классу: --mode spline для лого/иллюстраций, --mode polygon для иконок/пиксель-арта. Главные ручки: --filter_speckle (2–10), --color_precision (4–8), --corner_threshold.
5 · SVGO svgo --multipass — обязательный финиш.
6 · Валидация с условием остановки Проверяем SSIM(PNG↔рендер SVG) ≥ порога, число <path> ≤ бюджета, размер ≤ лимита. Превышение path при «vector-friendly» = сигнал «это было фото» → в ветку 2, а не бесконечный авто-сглаживатель. Жёсткий лимит итераций (например, 3) + fallback.
⚠️ Проверка фактуры: «Qwen 3.5 397B», «Qwen 3.6 35B-A3B» и «M5 Max» на дату задачи не существуют. Концепт проектировался по слухам о железе/моделях. Реальный план не должен от них зависеть — он и не зависит.
Первый шаг — в понедельник утром

brew install vtracer imagemagick && npm i -g svgo, киньте 20 своих реальных PNG в in/ и прогоните голый цикл vtracer --mode spline --color_precision 6 --filter_speckle 4 → svgo --multipass. Посмотрите глазами на 5 худших результатов. Это покажет, какая доля входа вообще векторизуема, и даст эталон, с которым потом сравнивать любой «умный» слой. Всё остальное — после.

Полные ответы советников

Контрарианец LLM здесь — паразит, а не мозг

Названные модели — фантомы. «Qwen 3.5 397B» и «Qwen 3.6 35B-A3B» на дату задачи не существуют; есть Qwen2.5-VL и Qwen3, но не в этих версиях/размерах. «M5 Max» тоже выдуман. Концепт построен на железе и весах, которых нет — проектирование по слухам.

Главный изъян — LLM здесь паразит. Классификация «логотип/фото/эскиз» решается гистограммой цветов и числом уникальных оттенков за 5 строк OpenCV, детерминированно. Vision-LLM на это — стрельба из гаубицы по воробью: недетерминированный, галлюцинирует «JSON-конфигом» с параметрами, которых не понимает физически.

Скрытые точки отказа: Real-ESRGAN перед трассировкой плодит артефакты-«мыло», VTracer обводит каждый, число path взрывается; цикл «повышаем сглаживание и повторяем» не имеет гарантии сходимости (логотип с градиентом → бесконечный цикл); фото в SVG — антипаттерн (200k path, файл тяжелее PNG); ComfyUI — нодовый GUI для диффузии, не пайплайн-движок.

Честный вопрос: что именно LLM делает, чего не делают 30 строк Python с детерминированными правилами? Если ответа нет — выкиньте Ollama, оставьте классификатор-эвристику + VTracer + SVGO.

Первопринципник Правильный вопрос — «что вообще стоит трассировать»

Трассировка фотографий бессмысленна — это главная ошибка концепта. SVG — формат для геометрии: контуры, заливки, кривые Безье. Фотография — поле непрерывных тонов без контуров. «Фото → SVG» даёт либо тысячи рваных полигонов (файл больше PNG), либо постеризованную кашу. Честный ответ для фото: <image> base64 либо отказ. Трассировка осмысленна только для геометричного: логотипы, иконки, штриховые эскизы, текст.

LLM-оркестратор — решение в поисках задачи. Качество SVG определяют три вещи: число цветов, контурность краёв, порог квантования. Всё меряется без нейросети: число уникальных цветов, доля краёв (Canny), энтропия гистограммы, доля прозрачности. Классификация — пороговый классификатор на 5–6 признаках.

Архитектура: 1) Триаж метриками OpenCV; 2) Фото → reject или <image>; 3) Vector-friendly → квантование → VTracer; 4) SVGO + валидация; 5) цикл оставить, но «слишком много путей» = сигнал «это было фото», а не «сгладь сильнее».

Чужак Начните с фильтра «годится / не годится», а не с «мозга»

Перевод PNG → SVG имеет смысл не для всех картинок, и это не проговорено. SVG выигрывает на простой графике (логотипы, иконки, схемы, шрифты): масштабирование, крошечный вес, редактируемость. На фото/градиентах — раздувается, тяжелее PNG, выглядит как пятна. Уже на входе нужен честный ответ: «эту оставьте PNG».

«Нейросеть-мозг» — переусложнение. Со стороны слышу: «не смогли подобрать настройки, поэтому повесили туда ИИ». У трассировки немного значимых ручек: цвета, сглаживание, удаление мусора — это пара эвристик. Локальная нейросеть тянет вес, время запуска, нагрев, непредсказуемость.

Что оттолкнуло бы: непонятно, что нажать и что получу (нужен превью «до/после» и размер); «проверка с повтором, если плохо» — кто судья «плохо»?; апскейл перед трассировкой = накопление ошибок. Начните с честного фильтра и трёх понятных ползунков.

Исполнитель Рабочий конвейер за один вечер, без LLM 
brew install vtracer imagemagick
npm i -g svgo

for f in in/*.png; do
  n=$(basename "$f" .png)
  vtracer --input "$f" --output "out/$n.svg" \
    --mode spline --color_precision 6 \
    --filter_speckle 4 --corner_threshold 60
  svgo --multipass "out/$n.svg"
done

Real-ESRGAN, ComfyUI, LangGraph, «Qwen-397B» — выкинь. Это даёт результат на 80% картинок.

Ручки VTracer: --filter_speckle (2–10, главный рычаг против тысяч path), --color_precision (4–8), --mode spline для лого / polygon для пиксель-арта; --gradient_step, --corner_threshold — потом.

Препроцессинг — только если грязно: magick in.png -fuzz 10% -trim +repage. Upscale/денойз — точечно. LLM появляется позже и не оркестратором: максимум мелкий классификатор (llava 7b) для пресета — 30 строк, на третьей неделе. 3 руками подобранных пресета бьют JSON от LLM. Валидация: размер + grep -c '<path', превысил порог — прогон с filter_speckle +2.

Экспансионист Батч — это сам продукт; стройте накапливающую систему

Батч — не «фича», это сам продукт. Если M5 Max и 128 ГБ простаивают, обрабатывать одну PNG за раз — преступление. Очередь на тысячи ассетов превращает «конвертер» в фабрику дизайн-ассетов.

Самое недооценённое — петля валидации. Есть числовая метрика качества (SSIM/perceptual diff между PNG и рендером SVG) — готовый сигнал для обучения. Логируй: класс → параметры VTracer → score. Через тысячу прогонов — датасет «параметр→качество», оркестратор начинает предсказывать. Само-тюнящийся пайплайн.

Смежное: дедуп иконок, нормализация палитры до токенов дизайн-системы, генерация вариантов (24/48px, тёмная тема), осмысленные имена. Это уже иконпак-генератор. И вектор наружу — обернуть в MCP-сервер: любой агент дёргает «векторизуй» как инструмент, локально и приватно.

Замечание председателя: ревью единогласно отметило это как слепое пятно — масштабировать и обучать имеет смысл только после рабочего ядра и честного триажа, иначе SSIM-петля закрепит фото-трассировку.
