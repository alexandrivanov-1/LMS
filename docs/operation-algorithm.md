# Алгоритм действий агента

## Когда пользователь даёт задачу по проекту (без указания на GitHub)
1. **Анализ запроса.** Внимательно читаю формулировку задачи, проверяю дополнительные инструкции (системные, пользовательские, `AGENTS.md`).
2. **Оценка текущего состояния репозитория.** Проверяю ветку, наличие незакоммиченных изменений, изучаю структуру кода и связанные документы.
3. **Планирование решения.** Определяю необходимые изменения, уточняю, нужны ли тесты, миграции, обновление документации.
4. **Реализация.** Вношу изменения в файлы, соблюдая стиль и инструкции.
5. **Локальная проверка.** Запускаю релевантные тесты, линтеры, проверки.
6. **Анализ результата.** Убеждаюсь, что все изменения соответствуют задаче и проходят проверки.
7. **Подготовка сообщения.** Готовлю структурированное резюме изменений и результатов тестов для пользователя.
8. **Коммит.** Фиксирую изменения с информативным сообщением.
9. **Подготовка PR.** Формирую PR-описание (через `make_pr`) с указанием изменений и тестов.
10. **Ответ пользователю.** Предоставляю подробный отчёт с ссылками на файлы и результаты команд.

## Когда пользователь просит действие на GitHub с использованием PAT
1. **Подтверждение инструкций.** Проверяю системные команды по настройке git, использованию PAT и ограничению remote.
2. **Проверка окружения.** Убеждаюсь, что переменные окружения `TOKEN`, `REPO_URL`, `AUTH_BASE` выставлены корректно, а `git` настроен на использование PAT только для репозитория `alexandrivanov-1/LMS`.
3. **Инициализация репозитория.** При необходимости клонирую репозиторий или обновляю текущую ветку согласно инструкциям пользователя.
4. **Выполнение требуемых действий.** Вношу изменения, выполняю команды GitHub (push, создание ветки, теги), строго следуя ограничениям хуков и конфигурации.
5. **Проверки.** Запускаю тесты или команды, связанные с задачей, удостоверяюсь в корректности результата.
6. **Коммит и push.** Формирую коммит, затем, если явно требуется, выполняю `git push` на `origin` (разрешённый remote). Если нужен PR, использую `make_pr`.
7. **Отчёт.** Подробно описываю в ответе все шаги, использованные команды, результаты тестов и подтверждаю, что PAT применялся только для целевого репозитория.

## Источники инструкций во время работы
- **Системные и пользовательские указания.** В первую очередь соблюдаю инструкции из текущего диалога и системных сообщений.
- **Файлы `AGENTS.md`.** Если в репозитории присутствуют такие файлы, то руководствуюсь их содержанием в соответствующей области проекта.
  - При старте новой задачи я проверяю, есть ли в целевой директории актуальные инструкции. Если ранее уже убеждался в их отсутствии и структура репозитория не менялась, повторный поиск не выполняю.
  - Такой поиск не является самодовлеющей процедурой, а лишь служит способом подтвердить, что действую в соответствии с последними инструкциями пользователя и спецификацией репозитория.
  - Эти файлы имеют особый приоритет, потому что в них владельцы проекта фиксируют обязательные правила: стиль кода, структуру каталогов, требования к коммитам и PR в отдельных частях дерева. Они считаются частью официальной документации и могут переопределять общие указания, поэтому их наличие нужно выявлять заранее.
- **Документация в репозитории.** При необходимости сверяюсь с профильными документами (`docs/` и другие каталоги), но не поддерживаю отдельный «секретный» файл с указаниями. Все применяемые правила задокументированы в репозитории или сообщены пользователем.
