# Чеклист полной синхронизации репозитория с веткой `main`

Документ помогает убедиться, что в GitHub-ветке `main` опубликована актуальная полная версия проекта LMS. Выполняйте пункты
последовательно, фиксируя результаты в issue или журнале задач.

## 1. Проверка локального состояния перед пушем
1. Убедитесь, что используете нужную ветку: `git switch main` или `git switch <feature-branch>`.
2. Проверьте, что нет неотслеживаемых файлов и незакоммиченных изменений: `git status`.
3. Убедитесь, что в индексе присутствуют все директории верхнего уровня проекта:
   - `frontend/`
   - `services/`
   - `infra/`
   - `docs/`
   - `db/`
   - прочие файлы (`README.md`, `AGENTS.md`, конфигурации CI и т.д.).
4. При необходимости добавьте отсутствующие файлы: `git add <path>` и создайте осмысленный коммит.

## 2. Синхронизация с удалённым репозиторием
1. Подтяните последние изменения `main`: `git fetch origin` и `git rebase origin/main` (или `git merge origin/main`).
2. Убедитесь, что удалённый `origin` указывает на `https://github.com/alexandrivanov-1/LMS.git` и использует PAT только для этого
   репозитория.
3. Перепроверьте историю коммитов: `git log --oneline origin/main..HEAD` должен быть пустым перед пушем в `main`.

## 3. Публикация изменений
1. Выполните пуш ветки: `git push origin <branch>`.
2. Откройте Pull Request в `main`, как описано в `/AGENTS.md` (ветка → PR → CI/Integration → squash-merge).
3. После squash-merge удалите временную ветку через интерфейс GitHub или командой `git push origin --delete <branch>`.

## 4. Пост-проверки на GitHub
1. Перейдите на страницу ветки `main`: <https://github.com/alexandrivanov-1/LMS/tree/main>.
2. Убедитесь, что в дереве файлов отображаются все ключевые директории и файлы, перечисленные в разделе 1. При необходимости
   сравните с локальным состоянием через `git ls-tree HEAD`.
3. В истории коммитов (`https://github.com/alexandrivanov-1/LMS/commits/main`) проверьте наличие вашего squash-коммита.
4. Убедитесь, что workflows CI и Integration на `main` завершились успешно.

## 5. Документирование результата
1. Обновите `docs/merge-report.md`, указав дату и ссылку на успешный пуш.
2. Добавьте запись в `docs/task-log.md` с описанием проделанной работы и перечислением изменённых путей.
3. Если использовались дополнительные инструменты (Codespaces, Actions artifacts), приложите ссылки в issue или PR.

## 6. Типичные ошибки и способы предотвращения
- **Недобавленные файлы.** Используйте `git status --short` и `git ls-files` для поиска пропущенных путей.
- **Ошибки аутентификации.** Не коммитьте PAT в репозиторий; храните его только в конфигурации Git.
- **Расхождения с `main`.** Перед пушем всегда делайте `git fetch` и пересобирайте ветку на свежем `origin/main`.
- **Неактуальная документация.** После публикации изменений убедитесь, что README и OpenAPI отражают текущую функциональность.

Следование этому чеклисту гарантирует, что ветка `main` на GitHub содержит полную и актуальную версию проекта.
